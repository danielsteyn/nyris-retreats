      Choose how guests book and pay when they click Reserve on a property page. You can switch any time.

      Stripe configuration

      The publishable key is safe to expose in the browser by design — that's how Stripe Elements works. Find it at Stripe Dashboard → Developers → API keys.

      The matching secret key (sk_…) belongs in Hospitable's Stripe Connect setup, never here.

      Hospitable Direct widget

      Paste the URL pattern Hospitable gave you for direct bookings. Use {propertyId} as a placeholder for the per-property ID — we'll substitute it on each Reserve click and append checkin, checkout, and guests as query params.

      If your account uses a different scheme (e.g. https://yourbrand.hospitable.com/{slug}), just paste it with the {propertyId} or {slug} placeholder where the per-property piece goes.

      Keeps your site open in the original tab. If unchecked, the Reserve button replaces the current page with Hospitable's booking page.

      Inline Hospitable widget on property pages

      When the provider above is set to Hospitable Direct widget, the per-property snippet you paste in Catalog → Property details replaces the default Reserve widget on that property's page (calendar, dates, payment all inline).

      Where to get the snippet (Hospitable's flow) Hospitable issues a unique snippet per property — there's no single template. In Hospitable: Direct Bookings tab → Website sub-tab → Create a site → choose "I already have a site" → check the property → three-dot menu → Copy widget code. Repeat per property. Paste each one into its own row in Catalog → Property details → "Hospitable booking widget snippet (this property)".

      Prerequisites: active Hospitable Direct plan, Stripe Connect linked, valid HTTPS site, payout method configured. The widget container reserves 320 × 900 px minimum (Hospitable's spec); the height auto-adjusts via postMessage resize events once their iframe announces its content height.

      Hospitable's embed is two parts: a shared listener script (here) and a unique iframe per property (in Property details). On each property page we render both, in order — listener first so it catches the iframe's resize postMessage events.
